### BUG-001

Название: Отсутствует интеграция между UI формой и API склада
Компонент: Integration
Severity: Critical
Priority: High

Предусловия: Открыта форма заказа, пользователь авторизован, на складе есть товар

Шаги:

Заполнить форму заказа (товар, количество)

Открыть DevTools → Network

Отправить форму

Проверить все исходящие запросы

Ожидаемый результат: После отправки формы должен быть запрос к Inventory API (/notes) для резервирования товара

Фактический результат: Отправляется только один запрос к бэкенду формы. Запрос к /notes отсутствует

Влияние: Резервирование не происходит, бизнес-требование не выполняется

Комментарий QA: Согласно DevTools, после submit'а формы нет никаких вызовов к practice.expandtesting.com/notes/api/notes

### BUG-002

Название: Модель Note не содержит поля для связи с заказом (orderId)
Компонент: API
Severity: Critical
Priority: High

Предусловия: Валидный токен авторизации

Шаги:

Вызвать GET /notes

Изучить структуру ответа

Ожидаемый результат: Модель содержит поле orderId или аналогичное для связи резервирования с заказом

Фактический результат: Модель Note содержит: id, title, description, category, completed, created_at, updated_at, user_id. Поля orderId нет

Влияние: Невозможно определить, какой заказ создал резервирование

Комментарий QA: Поле description используется для хранения количества, но это не позволяет связать запись с заказом

### BUG-003

Название: Отсутствует эндпоинт для возврата товара на склад при отмене заказа
Компонент: API
Severity: Critical
Priority: High

Предусловия: Существует резервирование (заметка) с количеством товара

Шаги:

Отменить заказ

Попытаться вернуть зарезервированное количество на склад

Ожидаемый результат: Существует API-метод для увеличения quantity (возврат товара)

Фактический результат: API поддерживает только PUT /notes/{id} (полное обновление) и PATCH /notes/{id} (обновление completed). Нет выделенного эндпоинта для инкремента количества

Влияние: При отмене заказа товар не возвращается на склад

Комментарий QA: Можно использовать PUT для обновления description (количества), но это требует знания текущего значения и не является атомарной операцией

### BUG-004

Название: API допускает создание заметки с отрицательным количеством
Компонент: API
Severity: Major
Priority: Medium

Предусловия: Валидный токен

Шаги:

POST /notes с description: "-5"

Проверить ответ

Ожидаемый результат: Ошибка валидации (количество не может быть отрицательным)

Фактический результат: Заметка создаётся успешно, description = "-5"

Влияние: Можно создать резервирование с отрицательным количеством, что нарушает бизнес-логику

### BUG-005

Название: Нет проверки наличия товара перед созданием заказа в UI
Компонент: UI
Severity: Major
Priority: Medium

Предусловия: Товара нет на складе (quantity = 0)

Шаги:

Заполнить форму заказа на товар, которого нет

Отправить форму

Ожидаемый результат: Пользователь видит сообщение «Товара нет в наличии»

Фактический результат: Форма принимает заказ (если поля валидны), показывается успешное создание

Влияние: Пользователь вводится в заблуждение — заказ создаётся, но товара нет

### BUG-006

Название: Отсутствует защита от race condition при параллельных заказах
Компонент: Integration
Severity: Critical
Priority: High

Предусловия: На складе 1 единица товара

Шаги:

Два пользователя одновременно отправляют заказ на 1 единицу

Проверить остаток

Ожидаемый результат: Только один заказ успешен, остаток = 0

Фактический результат: Текущая архитектура не имеет механизмов блокировки — оба заказа могут быть успешными

Влияние: Overselling — продажа товара, которого нет в наличии
